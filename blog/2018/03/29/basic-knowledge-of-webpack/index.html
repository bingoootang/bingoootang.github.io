# webpack 必知

随着前端技术的不断发展，前端开发工作从简单的网页开发逐渐变成了应用开发，而且应用的规模正在变得越来越大。随着应用规模的扩大，一个或者几个 js 文件再也无法承担所有的应用逻辑，因此，前端的模块化开发模式逐步被提了出来，并随之诞生了多个模块化规范，比如 CommonJS、AMD、ES6 等。
webpack 就是一个模块打包处理工具，帮助我们处理各种不同规范的模块，同时以模块的方式处理图片、字体等资源，最终打包成一组符合目标环境的分类资源。以浏览器为例，webpack 以模块（module）的方式将文件处理成 chunk，在处理过程中，通过 loader 加载不同类型的文件，同时提供若干钩子使得 plugin 可以对 chunk 进行处理，最后输出分类后的可直接在浏览器中运行的静态资源。

# 核心概念

  • entry 定义 webpack 处理的入口
  • loader 通过 loader 处理具体类型的文件，如 js、css等
  • plugin 通过 plugin 执行一些通用的任务，如代码压缩等
  • output 定义 webpack 如何输出处理后的文件
  • mode 定义运行模式，4.x 版本引入

# 输出结构

默认情况下，webpack 会将所有资源都打包成一个单独的 js 文件，将这个 js 文件通过 < script> 标签嵌入 html 文档中即可运行。但是，出于工程实际考虑，我们通常需要最大化利用客户端缓存，以缩短用户等待时间。此时，webpack 打包输出通常需要分成如下三个文件：

  • manifest/runtime
  • vendor
  • app

其中，manifest/runtime 叫做 webpack 的启动文件或者运行时，里面包含了 webpack 的启动和模块加载逻辑，没有这个文件，其他模块无法加载。

这这段代码中，window["webpackJsonp"] 的 push 方法被替换成了 webpackJsonpCallback，后面的模块通过向 window["webpackJsonp"] 内 push 自己完成加载。
manifest 的内容通常会改变，因此需要将这部分的代码独立出来，单独输出成文件。
vendor 当中保存了所有依赖包的代码，通常来自于 node_modules 目录，这部分代码的变更频次没有应用逻辑的变更频次高，因此可以单独出来，方便客户端缓存。
app 中保存应用的逻辑代码，这部分变更频次最高，因此需要单独输出，防止因为应用逻辑的变更导致依赖包的缓存失效。

# bundle vs chunk

看 webpack 的文档，一定会被各种术语搞晕，尤其是 bundle 和 chunk。从 webpack 文档的表述来看，bundle 和 chunk 仿佛都表示模块或模块的集合，那么他们有什么不一样呢？其实一个 bundle 就是一个 chunk，只是表述方式不同而已。在 webpack 内部，用 chunk 来进行表述，而社区一般更关注与输出，将其称为 bundle。从 webpack 团队核心成员的表述中，我们知道，chunk 是对 webpack 依赖图（dependency graph）部分的封装，通常分为三种:

  • Entry chunk
  • Initial chunk
  • Normal chunk

Entry chunk 由 webpack 的启动代码组成，比如上面的 menifest。Initial chunk 是除了 Entry chunk 以外的，必须被同步加载执行的模块，比如首屏业务代码。Normal chunk 与 Initial chunk 很相似，唯一的不同是 Normal chunk 通常是异步加载的，比如通过 require.ensure 指定的按需加载逻辑。
总之，chunk 是对源代码的一种抽象，它封装和定义了如何将源代码的 module 打包输出成文件。
